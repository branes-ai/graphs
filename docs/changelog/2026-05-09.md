# Changelog 2026-05-09

PhysicalSpec foundation + profile-as-SKU addressing arc: introduced a chip-level
metadata sibling to `HardwareResourceModel`, populated it for H100 + the four
Jetson SKUs, built the spec-sheet CLI on top of it, made power profiles
addressable as `<silicon>@<profile>` SKUs, sourced PhysicalSpec data live from
embodied-schemas YAMLs (with a bandwidth-math validator and KNOWN_OVERRIDES
table for documented YAML bugs), and surfaced per-profile memory clocks.

This work spans nine merged PRs (#133, #134, #135, #137, #138, #139, #141, #142,
#143) and resolves issue #131. Phases 1-4 of the umbrella issue #136 are
complete; Phase 5 lands separately in `branes-ai/embodied-ai-architect#201`.

## Added

- **`PhysicalSpec` dataclass** at `src/graphs/hardware/physical_spec.py`. Holds
  chip-level fabrication / spec-sheet metadata (die size, transistor count,
  process node, foundry, architecture, packaging, memory type + bus width,
  launch info) as a *sibling* to `HardwareResourceModel`. The mapping path
  (roofline / energy / scheduling) reads `resource_model` and never sees
  `physical_spec`; spec-sheet tools read `mapper.physical_spec` without
  burdening the hot path. PR #133.
- **`HardwareMapper.physical_spec` attribute**, initialized to `None` in the
  base `__init__`. Factory functions assign post-construction when data is
  available -- no subclass `__init__` changes needed. PR #133.
- **`cli/list_hardware_resources.py`** spec-sheet CLI. Registry-driven
  (walks `list_all_mappers()` / `get_mapper_by_name()`, picks up new mappers
  automatically -- 46 today vs 31 hand-listed in `list_hardware_mappers.py`).
  Four output formats: text (wide table), JSON, CSV, real Markdown (per-category
  tables). `--output FILE` auto-detects format from extension; `--sort {name,
  die_size, transistors, density, launch_year, tdp, tops_per_watt}`; `--reverse`;
  `--verbose` for per-product detail block. Renders `N/A` for unpopulated fields
  so the table is honest about partial coverage. Resolves #131. PR #134.
- **PhysicalSpec backfilled for 4 Jetson SKUs** (Orin AGX 64GB, Orin NX 16GB,
  Orin Nano 8GB, Thor 128GB). All three Orin variants share the GA10B die
  (455 mm², 17B transistors, Samsung 8N); only module-level fields differ per
  SKU. Thor's die size and transistor count are not publicly disclosed, so
  those fields stay `None`. New module-level helper `_orin_soc_physical_spec()`
  in `gpu.py` avoided literal duplication across the three Orin variants. PR #135.
- **Profile-as-SKU registry aliases** for power-modulated edge SoCs.
  `get_mapper_by_name` now resolves `{silicon}@{profile}` (recommended) and
  `{silicon}-{profile}` (dashed convenience form) alias names. New
  `list_all_skus(include_profile_aliases=True)` enumerates all 122 addressable
  SKUs (46 silicon-bins + 76 profile aliases). The `thermal_profile=` kwarg
  always wins over an alias-encoded profile. Backward-compatible: bare-name
  lookup with no kwarg still uses the default profile. Phase 1 of #136.
  PR #137.
- **Fourth canonical mode for Orin Nano**: added a distinct `25W`
  ThermalOperatingPoint (DVFS-on, "production deployment with explicit cap")
  alongside the existing `MAXN` (DVFS-off, "developer/burst, locked at boost").
  Both share the 25W TDP cap on Super silicon but differ in DVFS behavior.
  PR #137 (review-fix commit).
- **`--profiles {default, all}` flag** on `cli/list_hardware_resources.py`.
  Under `all`, expands multi-profile chips into one row per (silicon × profile)
  with the alias name (`Jetson-Orin-Nano-8GB@7W`); single-profile chips keep
  their silicon-bin row. Three new dataclass fields surfaced as columns:
  `mode`, `core_base_mhz`, `core_boost_mhz`. Phase 2 of #136. PR #138.
- **`memory_type` and `memory_bus_width_bits` on `PhysicalSpec`**. Module SKU
  constants that don't change per nvpmodel mode. Backfilled for H100 + 4 Jetsons
  with values cross-validated against bandwidth math (`BW = bus_width / 8 *
  DRAM_rate`). Phase 3 of #136. PR #139.
- **`physical_spec_loader` module** at `src/graphs/hardware/physical_spec_loader.py`.
  `load_physical_spec(base_id)` reads the embodied-schemas YAML database;
  `validate_physical_spec(...)` runs bandwidth-math + density-plausibility
  checks (5% tolerance for vendor rounding); `KNOWN_OVERRIDES` dict applies
  field-level corrections for documented YAML bugs (currently 2 entries from
  `branes-ai/embodied-schemas#8`). Path resolution: `EMBODIED_SCHEMAS_DATA_DIR`
  env var > installed `embodied_schemas` package > sibling-clone heuristic.
  Five factories converted to `load_physical_spec_or_none(base_id)` for
  graceful fallback when embodied-schemas data is unreachable. Phase 3.5 of
  #136. PR #141.
- **`memory_clock_mhz` field on `ThermalOperatingPoint`**. Per-profile because
  Jetson nvpmodel can throttle DRAM separately from compute clocks. Convention:
  internal DRAM clock (data rate / 2) -- LPDDR5-6400 = 3200, LPDDR5X-8533 =
  4267, HBM3-5230 = 2615. Populated for 15 Jetson thermal profiles (Orin family
  at 3200 MHz, Thor at 4267 MHz). Surfaced as the **Mem MHz** column in the
  spec-sheet view. Phase 4 of #136. PR #142.

  and `v4-validation` (in `v4-validation.yml`). Pinned to commit

  `--output FILE` flag with format auto-detection (`.json/.csv/.md/.txt`).
  Bundled with the PhysicalSpec foundation work. PR #133.

## Fixed

- **Two embodied-schemas YAML bugs caught via bandwidth-math sanity check**:
  - Jetson Thor 128GB: YAML lists 512-bit memory bus. Math:
    `273 GB/s / 8.533 GT/s LPDDR5X = 256 bits`. NVIDIA's Jetson Thor
    announcement blog explicitly states "256-bit LPDDR5X, 273 GB/s".
  - Jetson Orin Nano 8GB: YAML lists 64-bit memory bus. Math:
    `68 GB/s / 4.267 GT/s LPDDR5-4267 = 128 bits`. NVIDIA's Orin Nano
    datasheet specifies 128-bit.

  Filed as `branes-ai/embodied-schemas#8`. The `KNOWN_OVERRIDES` table in
  `physical_spec_loader.py` carries field-level corrections that retire
  automatically when the upstream YAMLs are fixed. The bandwidth-math
  `validate_physical_spec()` checker would have caught both at YAML import
  time -- it's now in place to prevent recurrence. PR #139, PR #141.
- **`cli/list_hardware_resources.py` Node column overflow**. The fixed-width
  `>10` Node column overflowed on `"Samsung 8LPP"` (12 chars), pushing every
  column after it right and breaking row alignment. Widened to `>14` and
  truncated values to 13 chars. PR #143.

# Changelog 2026-05-12

Three merged PRs across two repositories complete the KPU power-modeling and
catalog-naming arc that began with the t256 PE-array discontinuity flagged in
the prior session. TDP is now derived from `(clock, Vdd, ProcessNode energies,
WorkloadAssumption)` with V²·f scaling instead of being architect-declared, and
the catalog adopts a self-describing SKU naming convention
(`kpu_t<count>_<rows>x<cols>_<mem><ch>_<value>nm_<foundry>_<library>`) that
makes "vary one variable at a time" comparison sweeps query-able via three new
`list_kpus.py` filter flags.

The catalog grew from 4 SKUs to 12: 4 legacy renames (`stillwater_kpu_t*` →
new convention), plus 8 new sweep variants spanning t64/t128/t256/t512 across
GF 12FDX (FD-SOI, energy-efficient) and TSMC N7 HPC (datacenter compute) at
identical operating points, so cross-foundry deltas are isolatable.

Landed via:
- **embodied-schemas PR #14** (squash `846a0d2`) — naming convention + 12FDX/N7 sweeps + legacy migration
- **graphs PR #153** (squash `3549472`) — derived TDP from V/f/workload + PE-array sweep CLI
- **graphs PR #155** (squash `e8e2463`) — adopt new KPU SKU naming convention; rename legacy refs

Plus tracking issue **graphs#154** for the leakage(Vdd) follow-up.

## Added

### KPU power model (graphs PR #153)

- **`graphs.hardware.kpu_power_model`** new module
  (`src/graphs/hardware/kpu_power_model.py`, 315 lines) that derives per-thermal-
  profile TDP from chip configuration. Five-term roll-up: PE compute (uses the
  existing `ProcessNode.energy_per_op_pj` lookup), L2 SRAM access, L3 SRAM access,
  NoC traversal, DRAM PHY, plus chip leakage. V²·f scaling on all dynamic terms
  via `(profile.vdd_v / node.nominal_vdd_v)²`. `WorkloadAssumption` dataclass
  exposes the workload knobs (compute_duty_cycle=0.15, bytes_per_op=1.5,
  L1/L2/L3 hit rates 0.95/0.85/0.70, NoC traversal ratio). Picks the worst-case
  precision per profile (turns out to be bf16 for the current SKU set since
  1.4 vs 0.3 pJ/op outweighs the lower ops/clock).
- **`apply_pe_array_override(spec, rows, cols)`** helper in
  `graphs.hardware.kpu_sku_generator` for roadmap-style PE-array sweeps.
  Resizes every tile class's PE array and rescales `ops_per_tile_per_clock`
  by `new_pes/old_pes` so per-PE op throughput is preserved; pipeline
  fill/drain cycles track `max(rows, cols)` per the family convention.
  silicon_bin coefficients untouched (`kind=per_pe` blocks auto-scale).
- **`cli/generate_kpu_sku.py --pe-array ROWSxCOLS`** CLI flag wires the
  override into the existing `--from-sku` flow:
  `--from-sku kpu_t256_..._16nm_tsmc_ffp --pe-array 16x16 --output ...yaml`.
- **`cli/show_compute_product.py`** new inspector (296 lines) that prints
  the assembled ComputeProduct hierarchy for one KPU SKU: the SKU layer +
  referenced ProcessNode + every CoolingSolution referenced by a thermal
  profile, plus a thermal-headroom cross-check panel that flags TDP-vs-cooling
  and power-density-vs-cooling violations. Reuses per-layer renderers from
  `show_kpu.py` / `show_process_node.py` / `show_cooling_solution.py` so
  formatting stays consistent. Auto-detects `--output` extension across
  JSON / CSV / MD / text per the CLI rule.

### KPU catalog (embodied-schemas PR #14)

- **8 new sweep SKUs** under `data/kpus/stillwater/`:
  - GF 12FDX (energy-efficient): `kpu_t{64,128,256,512}_32x32_lp5x{4,8,16,32}_12nm_gf_fdx`
  - TSMC N7 HPC (datacenter): `kpu_t{64,128,256,512}_32x32_lp5x{4,8,16,32}_7nm_tsmc_hpc`
  - Both sweeps share architecture (32×32 PE arrays, 70/20/10 INT8/BF16/Matrix
    tile mix, lpddr5x with channel count scaling 4/8/16/32) and operating
    points (lp/default/boost at 450/600/700 MHz × 0.60/0.75/0.90 V on N16
    style; 0.55/0.70/0.85 V on N7 to fit Vnom=0.75 floor) so the only
    variable changing between sweeps is the process node.
  - Cross-foundry comparison validation: at fixed (256-tile, 32×32 PE,
    600 MHz @ 0.65 V), 12FDX gives 22 W / 214 mm² / 315 INT8 TOPS and
    N7 HPC gives 10 W / 59 mm² / 315 INT8 TOPS — identical compute at
    half the power and 3.6× smaller die.
- **GF 12FDX node energies** populated in
  `data/process-nodes/globalfoundries/12fdx.yaml`:
  `sram_access_pj_per_byte` (per CircuitClass), `dram_io_pj_per_byte` (PHY-side
  LPDDR PHY), `noc_pj_per_flit_per_hop` (per router library). Cites GF 12FDX
  SRAM characterization and standard LPDDR5 PHY references.

### `list_kpus.py` filtering (graphs PR #155)

- **Three new filter flags**, AND-combinable with existing `--vendor` /
  `--target-market` (`cli/list_kpus.py`):
  - `--foundry <name>` — exact match on `ProcessNodeEntry.foundry`, with
    short-form aliases (`gf` → `globalfoundries`, `sec` → `samsung`)
  - `--node-nm <int>` — exact match on `ProcessNodeEntry.node_nm`
  - `--library <tag>` — substring match against the LAST `_`-segment of the
    SKU id (the library tag's position in the new naming convention) so
    `--library lp` doesn't accidentally match `lp5x16` memory tokens
- **Dynamic column widths** for `id`, `node`, and `tier` columns in the text
  renderer so long-form SKU ids (~30-35 chars) and foundry-prefixed node ids
  no longer break alignment.
- **`KPURow` gains `foundry` and `node_nm` fields** populated from the
  resolved ProcessNode (also exported in JSON / CSV / MD output).

## Changed

### Schema additions (embodied-schemas PR #14, on top of PR #13's prior session work)

- **`KPUThermalProfile.vdd_v: Optional[float]`** — per-profile core supply
  voltage. Default `None` = use `ProcessNode.nominal_vdd_v`. Lets architects
  express Orin-style DVFS where lower-power modes drop both clock AND voltage
  so the V²·f product spreads TDP across profiles.
- **`KPUThermalProfile.activity_factor: Optional[float]`** — per-profile
  multiplier on the workload duty cycle. Tunes individual operating points
  without changing the chip-wide workload model.
- **`ProcessNodeEntry.sram_access_pj_per_byte: dict[CircuitClass, float]`** —
  per-byte L2/L3 access energy by SRAM library. L1 stays rolled into
  `energy_per_op_pj` for the PE library.
- **`ProcessNodeEntry.dram_io_pj_per_byte: Optional[float]`** — PHY-side
  DRAM I/O energy at the chip package boundary.
- **`ProcessNodeEntry.noc_pj_per_flit_per_hop: dict[CircuitClass, float]`** —
  per-flit-per-hop NoC traversal energy by router library.

### Generator behavior (graphs PR #153)

- **`generate_kpu_sku()` now derives TDP** instead of trusting the architect's
  hand-authored `tdp_watts`. For each input thermal profile, computes TDP via
  the kpu_power_model and writes the derived value into the output. Default
  TDP / max / min derived from the per-profile derived values.
- **Catalog regen for all 4 legacy SKUs** with derived TDPs landing within
  ±0.5 W of catalog targets (3W/6W/10W on t64; 6W/12W/18W on t128;
  15W/30W/50W on t256; 30W/60W/100W on t768). Required dropping catalog
  clocks substantially (e.g., t256 default 1400 → 600 MHz) and adding
  per-profile Vdd values, since the prior catalog clocks were aspirational
  marketing numbers — chips couldn't physically sustain them at the
  advertised TDPs under the workload model. New clocks line up with
  real-world Orin AGX (420/624/828 MHz at 15/30/50 W).

### Naming convention (embodied-schemas PR #14, graphs PR #155)

- **All KPU SKU ids** migrate to
  `kpu_t<count>_<rows>x<cols>_<mem><ch>_<value>nm_<foundry>_<library>`. Library
  tag conveys product positioning: `fdx` / `ull` (energy-efficient FD-SOI),
  `hpc` (datacenter compute), `lpp` / `pp` / `ffp` (FinFET performance), `lp`
  / `hd` (mainstream / high-density). Vendor encoded in the directory path
  (`data/kpus/stillwater/`) rather than the filename.
- **Legacy renames**:
  - `stillwater_kpu_t64` → `kpu_t64_32x32_lp5x4_16nm_tsmc_ffp`
  - `stillwater_kpu_t128` → `kpu_t128_32x32_lp5x8_16nm_tsmc_ffp`
  - `stillwater_kpu_t256` → `kpu_t256_32x32_lp5x16_16nm_tsmc_ffp`
  - `stillwater_kpu_t768` → `kpu_t768_16x8_hbm3x16_7nm_tsmc_hpc`
- **Tier vocabulary expanded** from 4 buckets to 5:
  `entry < mid < high < enthusiast < datacenter`. The new t512 SKUs are
  `enthusiast` — matches Intel Core X / AMD Threadripper / NVIDIA RTX Spark
  prosumer positioning.
- **Hardware registry** parallel-store JSON specs in
  `hardware_registry/accelerator/` renamed to match the new ids (directory
  name + `spec.json` `id` field updated).
- **All graphs source touched**: factory modules
  (`kpu_t{64,128,256,768}.py`), mapper (`mappers/accelerators/kpu.py`
  `base_id`s), loader (`kpu_yaml_loader.py` controller-count map keys),
  validator references (`sku_validators/{__init__,context}.py` examples),
  and CLI tools (`show_*.py`, `validate_sku.py`, `generate_kpu_sku.py`
  default/example args). Factory function names kept short
  (`kpu_t64_resource_model()` etc.) and mapper display names
  (`Stillwater-KPU-T64`) preserved as user-facing labels.

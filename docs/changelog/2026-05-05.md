# Changelog 2026-05-05

This session landed the V4 model-validation harness end-to-end (Phase A of
V4-4 + V4-5 + V4-6) and addressed two of the architectural-model bugs the
harness surfaced (#71, #61).

## Added
- **V4-4 Phase A: NVIDIA + Jetson ground-truth measurers** (#80) -- adds
  `validation/model_v4/ground_truth/pytorch_cuda.py` (cudaEvent + NVML
  with total-energy / power-fallback paths) and
  `validation/model_v4/ground_truth/pytorch_jetson.py` (cudaEvent +
  INA3221 sysfs, two-layout detector for JetPack 5+ hwmon and JetPack 4
  iio). Plus `validation/model_v4/sweeps/_augment.py` for additive
  re-classification of existing sweep entries against new hardware
  targets without churning committed JSONs. 16 files, 2375 insertions,
  29 new tests.
- **Capture-on-target runbooks** (`bin/v4-capture-jetson-orin-nano.sh`,
  `bin/v4-capture-h100.sh`) -- checkpoint-based runbooks shipped
  alongside the measurers in PR #80.
- **V4-5 KPU consistency-only invariant suite** (#82) -- new
  `validation/model_v4/invariants/kpu.py` with 6 hard invariants
  (roofline self-consistency, monotonicity, achieved-below-peak, family
  monotonicity) + 1 documented-xfail diagnostic (avg-power-below-TDP,
  tracked in #81). 583 lines, 30 tests + 4 xfails.
- **V4-6 dedicated CI gate workflow** (#83) -- new
  `.github/workflows/v4-validation.yml` that surfaces V4 failures as a
  distinct "V4 Model Validation" check in the PR status rollup, with a
  path filter so it only runs on PRs that could shift the analytical
  model. Plus 6 meta-tests pinning workflow structure and a new
  `validation/model_v4/README.md` contributor guide.
- **On-chip BW peaks + L1_BOUND classifier dispatch** (#84, resolves
  #61) -- adds three Optional float fields to `HardwareResourceModel`
  (`l1_bandwidth_per_unit_bps`, `l2_bandwidth_bps`, `l3_bandwidth_bps`),
  populates the V4 reference targets (i7-12700K, H100-SXM5, KPU-T64)
  with cited values, and teaches `validation/model_v4/sweeps/classify.py`
  a bandwidth-aware dispatch path that emits a new `Regime.L1_BOUND`.
  12 new tests; capacity-only path preserved verbatim for unaugmented
  mappers.

## Fixed
- **CPU energy under-prediction** (#77, resolves #71) --
  `EnergyAnalyzer.CPU_IDLE_POWER_FRACTION` 0.1 -> 0.7 with reinterpreted
  semantic ("average package power during active execution as fraction
  of TDP"), calibrated against V4 RAPL baseline. Plus sub-ms RAPL
  reliability skip in `validation/model_v4/harness/assertions.py`. V4
  i7-12700K matmul `pass_energy` 0/60 -> 12/60 with 30 sub-ms shapes
  cleanly skipped.
- **CI gate import-path failure** -- `.github/workflows/v4-validation.yml`
  needed `PYTHONPATH=${{ github.workspace }}` so `from validation.model_v4...`
  imports resolve in the gate's narrow pytest invocation.
- **CI gate token permissions** -- added explicit `permissions: contents: read`
  to the V4 gate workflow per CodeQL alerts #11/#12.

## Infrastructure
- 4 sweep JSONs re-augmented additively with H100 and 3 Jetson keys
  (i7 entries unchanged, byte-for-byte).
- `validation/model_v4/harness/runner.py` `SWEEP_HW_TO_MAPPER` extended
  with the 4 new hardware keys.
- `validation/model_v4/cli/capture_ground_truth.py` `_MEASURER_FACTORY`
  dispatches to the new backends via lazy import (so the i7 dev box
  without pynvml/torch.cuda can still import the CLI module).

## Issues filed (deferred follow-ups)
- **#78** -- `EnergyAnalyzer` per-unit CPU allocation underestimates
  uncore/IMC power. Real concern but V4 harness uses the no-allocation
  fallback so V4 numbers are unaffected.
- **#79** -- `EnergyDescriptor.confidence` defaults to UNKNOWN even
  when the energy model is calibrated. Systemic gap across all
  estimators; warrants a focused PR.
- **#81** -- KPU energy model uncalibrated; predicted avg power
  exceeds TDP by ~1.5x-3x on every KPU mapper. Same shape of bug as
  #71 was for CPU. The V4-5 `avg_power_below_tdp` invariant is xfail
  with this as the tracking issue.
